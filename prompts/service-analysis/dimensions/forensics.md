# Dimension: Forensics — "왜 이렇게?" (항상 마지막)

## Intent

앞선 차원에서 수집된 **개별 의문점**들을 **큰 그림 가설**로 통합한다.
이게 바로 이 서비스를 "코드로만" 파악하는 것과 **"설계자의 머릿속에 들어가는"** 것의 차이.

이 섹션이 잘 채워지면 나중에:
- 여러 서비스를 비교할 때 팀의 **설계 일관성/불일치**가 드러남
- `wiki reflect` 가 공통 패턴을 찾아 새 dimension을 제안
- 팀 온보딩 시 **"왜 이 코드가 이런 모양인지"** 대화의 출발점

## What to look for

앞선 차원에서 수집된 모든 "❓ 의문점"을 다시 훑고, 다음을 시도한다:

1. **패턴 발견**: 여러 차원에서 같은 외부 시스템 / 같은 도메인 엔티티 / 같은 실패 모드가 반복 등장하는가?
2. **상충**: 어떤 차원은 강한 일관성을 추구하고(분산락), 다른 차원은 포기(AFTER_COMMIT 이벤트 누락 허용) — 의도적 트레이드오프인가 레거시 혼재인가?
3. **진화 흔적**: `@Deprecated`, 주석의 "TODO / FIXME / 2024-xx에 교체 예정", 사용되지 않는 클래스 — **과거에서 현재로 가는 방향**.
4. **정치**: 특정 다른 서비스(`order-service` 등)를 직접 DB로 읽는 곳이 많으면 → **팀 간 경계 미성숙** 신호.

## Output section template

```markdown
## ❓ 설계 의문점 / "왜 이렇게?" (포렌식)

### 1. <가장 큰 의문 제목>

- **관찰**: (앞 차원에서 근거 인용) "A 엔티티가 B DB에 저장되고 있고, C 플로우는 분산락 + DB 락을 이중으로 씀"
- **가설 A**: 과거 장애로 이중 방어가 추가됐을 가능성
- **가설 B**: 분산락 leaseTime 값에 확신이 없어서 DB 락이 안전망
- **확인 방법**:
  - `git log -- **/ClaimRepository.kt` 에서 PESSIMISTIC_WRITE 추가 커밋
  - 팀 Jira / 장애 리포트
  - `@SchedulerLock` 값 결정 PR의 review 코멘트

### 2. <두 번째 의문>

...

### 3. <세 번째 의문>

...

### 종합 가설 (선택)

이 서비스를 한 줄로 표현하면:
> "외부 서비스 데이터를 read-only로 흡수해 claim 도메인을 주도하되,
>  결제·배송과의 상호작용은 이벤트 기반으로 분리해 원자성보다 복구 가능성을 택한 서비스."

— 이 한 줄이 맞는지 검증 가능한 질문 2~3개를 남겨둔다.
```

## Forensic prompts (메타)

- 앞 차원들에서 공통적으로 **외부 시스템 X**가 문제의 원인으로 언급되는가? → X에 대한 별도 단락 필요
- **트레이드오프의 방향**이 일관적인가? (예: 항상 일관성보다 가용성 선호 / 항상 원자성 추구)
- **"이 코드는 왜 존재하는가"** 를 3번 꼬리 물어 물으면 결국 비즈니스 요구로 닿는가, 아니면 **"과거의 기술 선택"** 에서 끝나는가?

## When absent

이 차원은 절대 스킵 금지. 내용이 부족하면 "의문점 2개 + 추가 조사 필요" 로라도 기록.
